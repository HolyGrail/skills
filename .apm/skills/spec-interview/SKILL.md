## Spec Interview

既存の spec ファイルを読み込み、**AskUserQuestionTool を使った深掘りインタビュー**で仕様を完全に肉付けする。自明な質問は排除し、実装上のトレードオフ・エッジケース・非機能要件の具体的閾値など、設計判断に直結する質問のみを行う。インタビュー完了後、収集した情報を元の spec ファイルに書き込む。

### 使い方

```bash
/spec-interview [ファイルパス]
```

- 引数なし: カレントディレクトリの `SPEC.md` を読み込む
- 引数あり: 指定ファイルを読み込む（例: `/spec-interview docs/design/SPEC.md`）

### Claude が守るべき行動指針

#### 必須ルール

1. **必ず AskUserQuestionTool を使う** — テキストで質問を羅列するのではなく、毎回 `AskUserQuestionTool` を呼び出して選択肢付きの質問を提示する
2. **1 回に 2〜4 問** — 一度に大量の質問をせず、回答を受けてから次の深掘りをする
3. **自明な質問を禁止** — 「何を作りますか？」「ユーザーは誰ですか？」のような、spec を読めば分かる質問は絶対にしない
4. **回答を踏まえた follow-up** — 各回答に対して「その選択をした場合の具体的な問題」を掘り下げる
5. **完全になるまで継続** — インタビューを途中で切り上げず、全カテゴリが網羅されるまで続ける
6. **最後に必ず書き込む** — インタビューが完了したら、Write ツールで spec ファイルを更新する

#### 質問の質の基準

**✅ 良い質問の例**
- 「A と B のアプローチを比較したとき、あなたが優先したいのはどちら？その理由は？」
- 「〇〇 が失敗したとき、ユーザーはどのような状態になることを許容できる？」
- 「レスポンスタイムの上限は具体的に何 ms？99 パーセンタイルで？」
- 「1 年後にこの機能をどう変える想定？その拡張を今から考慮すべき？」
- 「A を優先すると B のパフォーマンスが 30% 落ちる可能性がある。それでも OK？」
- 「実装で一番難しいと予想している部分はどこ？」

**❌ 悪い質問の例**（禁止）
- 「この機能の目的は何ですか？」（spec に書いてある）
- 「ユーザーは誰を対象にしますか？」（spec に書いてある）
- 「どのような機能が必要ですか？」（spec に書いてある）
- 「認証は必要ですか？」（自明すぎる）

### インタビューフェーズ

#### Phase 1: Spec 読み込みと分析

1. Read ツールで spec ファイルを読む
2. 以下の観点で分析し、不足・曖昧な箇所をリストアップ:
   - 技術的決定が未確定な箇所
   - 競合するアプローチが存在する箇所
   - エラーケース・失敗シナリオが未定義な箇所
   - 非機能要件（パフォーマンス・セキュリティ・スケール）の具体的数値が不明な箇所
   - 将来の拡張性と現在の制約のバランスが未決な箇所
   - ステークホルダー間の優先順位矛盾が潜在する箇所

#### Phase 2: 深掘りインタビュー（ループ）

以下のカテゴリを網羅するまでインタビューを繰り返す。**各ラウンドで 2〜4 問**を AskUserQuestionTool で提示し、回答を受けて次のラウンドに進む。

**必須カテゴリ（全て確認すること）**

- **技術アーキテクチャ**: ライブラリ・フレームワーク選択のトレードオフ、データモデルの具体的構造、API 設計の詳細
- **エラー・失敗シナリオ**: 各操作の失敗時の挙動、部分的失敗のハンドリング、リトライ戦略
- **非機能要件の数値**: レイテンシ閾値、スループット要件、可用性 SLA、データ保持期間
- **セキュリティ**: 認証・認可の詳細、機密データの扱い、脅威モデル
- **スケーラビリティ**: 現在の想定規模、将来の成長率、ボトルネック予測
- **UI/UX の具体的挙動**: 非同期処理中の UI 状態、エラー表示の詳細、モバイル・アクセシビリティ対応
- **データの整合性**: 競合状態のハンドリング、トランザクション境界、キャッシュ戦略
- **運用・監視**: ログに含める情報、アラート条件、デプロイ戦略
- **依存関係とリスク**: 外部サービス依存のリスク、最も不確実な部分、想定外のシナリオ

**AskUserQuestionTool の使い方**

各質問では必ず以下の形式を心がける:
- 選択肢は「具体的な技術名・数値・設計パターン」を提示する（「その他」は必要に応じて）
- 選択肢の説明でトレードオフを簡潔に示す
- 「Other」で自由記述を許容する

#### Phase 3: 完了判定

以下の全条件を満たしたらインタビュー完了と判断する:

- [ ] 全必須カテゴリをカバーした
- [ ] 主要なエラーシナリオが定義された
- [ ] 非機能要件に具体的な数値がついた
- [ ] 技術的な重要決定が確定した
- [ ] 未解決の曖昧さが spec に影響しない軽微なものだけになった

#### Phase 4: Spec 書き込み

収集した情報をもとに spec ファイルを更新する:

1. 既存の spec 内容を保持しながら、インタビューで得た情報を追加・補完
2. 以下のセクションを充実させる:
   - 技術スタック・アーキテクチャ決定とその根拠
   - エラーハンドリング仕様（具体的なエラーケースと対応）
   - 非機能要件（数値付き）
   - セキュリティ要件
   - データモデル・API 詳細
   - 未解決の課題・将来の拡張候補
3. Write ツールでファイルに書き込む

### 実行フロー例

```
/spec-interview

→ SPEC.md を Read で読み込む
→ 分析: 「認証方式が未定義」「エラー時の挙動が不明」「レイテンシ要件なし」等を特定

[Round 1] AskUserQuestionTool:
  - Q1: キャッシュ戦略（Redis vs in-memory vs なし）のトレードオフをどう判断する？
  - Q2: 外部 API が応答しない場合、ユーザーには何秒待たせる？タイムアウト後は？

[ユーザー回答]

[Round 2] AskUserQuestionTool（回答を踏まえた深掘り）:
  - Q1: Redis を使う場合、キャッシュの整合性はどのタイミングで保証する？
  - Q2: タイムアウト後のフォールバックに古いデータを返す場合、その「古さ」の許容範囲は？
  - Q3: この機能で最初にリリースする MVP の範囲と、後回しにする機能は？

... （完了まで継続）

→ 全カテゴリ完了
→ Write ツールで SPEC.md を更新
→ 「インタビュー完了。SPEC.md を更新しました。」
```

### 注意事項

- **前提条件**: 対象ファイルが存在すること（空ファイルでも可）
- **制限事項**: AskUserQuestionTool の選択肢は最大 4 つ。複雑な選択は複数ラウンドに分割する
- **推奨事項**: インタビューの途中でユーザーが「終わりにして」と言ったら、それまでの情報で spec を書き込む
- **エラー処理**: ファイルが見つからない場合は、ユーザーにファイルパスを確認してから進む
