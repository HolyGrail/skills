Spec レビューセッションを実施する。spec ファイルを読み込み、曖昧さ・欠落・未定義箇所をすべて洗い出したうえで、作者に対して **あらゆる詳細** について質問を重ね、spec が完全かつ曖昧さのない状態になるまでインタビューを継続する。命名、デフォルト値、エラーメッセージ、エッジケース、フィールド型、メソッドシグネチャ、戻り値、ログペイロード、設定キー、テスト期待値まで踏み込んで確認する。実装者がこの spec だけを読んで追加質問なしに実装できる状態に到達するまで止めない。

## 入力

ユーザーは引数として spec ファイルパスを指定する: $ARGUMENTS

引数がなければ、どの spec ファイルをレビューするかユーザーに確認する。

## 準備フェーズ（Preparation Phase）

質問を開始する前に必ず以下を実施する:

1. **Spec ファイルを精読** — 全文を Read ツールで読み込む
2. **参照ファイルをすべて読む** — spec 内で参照されているファイル（スキーマ、既存コード、設計ドキュメント）をすべて開き内容を把握する
3. **プロジェクト規約を確認** — `CLAUDE.md`（プロジェクト / global `~/.claude/CLAUDE.md`）、スタイルガイド、アーキテクチャドキュメント、knowledge ファイルなど、spec のドメインに関連する規約をすべて読む
4. **スタック固有の設定ファイルを特定** — `package.json` / `wrangler.jsonc` / `.dev.vars` / `Gemfile` / `mise.toml` / `Makefile` など、プロジェクトの技術スタックを示すファイルから既存の技術選定・命名規約・ディレクトリ構造を把握する
5. **ロードマップを確認** — 大局的なドキュメント、タイムライン、マイルストーン計画など、spec のドメインに影響する将来の変更がないか確認する
6. **既存コードベースを探索** — spec の実装が依存・連携する既存モデル、モジュール、パターン、命名規約を把握する
7. **外部ライブラリ仕様を context7 で取得** — spec が特定のライブラリ・フレームワーク・SDK に依存している場合、`mcp__plugin_context7_context7__query-docs` で最新のドキュメントを取得してから質問を組み立てる（学習データの記憶ではなく現行仕様に基づいて選択肢を提示するため）
8. **矛盾・バグのチェック** — スキーマ内のフィールド衝突、不適切なインデックス型、計算式の誤り、重複セクション、spec とコード間の命名不一致などを洗い出す

## 低摩擦原則（Low-Friction Principle）

準備フェーズで確認した既存のプロジェクト規約・コードパターン・スタイルガイドから**自明に導出できる**判断（命名規約、ファイル配置、既存のエラーハンドリングパターンへの追随など）は、インタビューで質問せず採用する。質問するのは以下の場合に限る:

- 既存規約と候補アプローチが衝突する
- 既存規約では扱われていない新しい設計判断が必要
- トレードオフが存在し、spec 作者の価値判断が必要

## インタビューフェーズ（Interview Phase）

`AskUserQuestion` ツールで **2〜4 問のバッチ** に分けて質問する。関連する質問をまとめる。**各バッチの回答を受け取るたびに、次のバッチに進む前に spec ファイルを更新する。** 長時間のインタビューでもコンテキストを失わず、spec が常に最新状態で進行するようにする。

インタビューは長丁場（多くの場合 10 ラウンド超）になる前提で臨み、焦らず・省略せず全領域を網羅する。後半の回答から前半の曖昧さが新たに見えた場合は、躊躇せず戻って再質問する。

**プロジェクト固有の規約（CLAUDE.md、スタイルガイド、knowledge）とフレームワーク固有の skill は常に最優先。** 以下のチェックリストは準備フェーズで発見した実際のスタック・パターンに合わせて調整すること。

### データモデル
- スキーマの正しさ（型、命名、制約、フィールド番号）
- DB スキーマの正しさ（インデックス、制約、カラム型、デフォルト値）
- シリアライゼーション方針
- リソース識別子パターン（URL/ID 設計）

### アーキテクチャ & 設計
- クラス・モジュール構造（ロジックの置き場所、組織化方法）
- インターフェース設計（メソッドシグネチャ、戻り値型、エラーハンドリング）
- 命名規約（クラス、メソッド、ファイル）
- 拡張パターン

### 振る舞い & ロジック
- ハッピーパスのエンドツーエンドフロー
- エラー条件それぞれの扱い
- エッジケース（並行性、競合状態、nil/null 値）
- バリデーションルールとその適用箇所

### インテグレーション
- 新規コードと既存モデル・モジュールの連携
- 設定値とデフォルト
- 暗号化・シークレット要件（Workers の `.dev.vars` / Secrets Store、環境変数管理など）
- バックグラウンドジョブ設計
- ログ計画（どのイベントをどのレベルで、どのペイロードを出すか）

### セキュリティ
- 資格情報の扱い（ハッシュ化、比較、保管）
- リプレイ攻撃対策
- レートリミット戦略とデフォルト値
- ログに出してはいけない機密データの明示

### 運用
- 開発セットアップとフィクスチャ
- パージ・クリーンアップ戦略
- キーローテーション対応

### 将来への備え（Future-proofing）
- 予見可能な変更（ロードマップ、タイムライン、作者の知識に基づく）に現設計が耐えられるか
- 作者が意識的に受け入れているリスク — 実装者と将来の読者がトレードオフを理解できるよう明示的に記録する
- 既知の今後の変更がこの機能に影響する場合のマイグレーション・進化パス
- 将来変更される可能性のあるインターフェース・データモデルが、非破壊的に拡張可能な設計になっているか

### スコープ & 成果物
- 明示的な in/out of scope
- 将来作業への依存関係
- 期待されるファイル成果物（コード、spec、ドキュメント）
- ドキュメント成果物ごとに、何をカバーし、何をソースファイルへの参照で済ませるか

## ルール

- **絶対に仮定しない** — 曖昧な箇所は必ず質問する。答えが分かる気がしても、ユーザーに確認する
- **低摩擦原則と矛盾しない** — 既存規約から自明な箇所は質問しない。質問対象は「曖昧」「規約が存在しない」「価値判断が必要」のいずれかに該当する項目に限定する
- **バグを報告** — 参照コードやスキーマにバグを見つけたら、質問の前にまず指摘する
- **具体的に訊く** — 質問には具体的な選択肢と説明を添える。選択肢を提示できる場面でオープンクエスチョンを投げない
- **逐次更新** — 2〜4 問の回答を受けたら、次のバッチに進む前に必ず spec 本文と Current Status セクションを更新する。長時間インタビューではこれが必須。未適用の回答を 1 バッチ分以上ためない
- **文章を磨く** — spec を更新する際は、追加した箇所だけでなく周辺の文章も読み返し、ぎこちない表現・冗長な文を整える。インタビュー回答を継ぎ接ぎした文書ではなく、一貫したトーンで読める設計書にする
- **Spec の自己完結性** — spec は単独で成立する文書でなければならない。設計詳細、スキーマ、決定の根拠はすべて spec 本文にインライン化する。読者が他ファイルを開かなければ全体像が掴めない状態にしない
- **TODO の掃討** — spec 内のすべての TODO を見つけて解決する。成果物として残す TODO（実装時にファイルを作成する等）は、質問ではなく明確なアクションアイテムとして書き直す
- **完了確認** — 最後に TODO と曖昧表現（"TBD"、"maybe"、"possibly"、"to be decided"、「検討中」「たぶん」「要検討」など）が残っていないか grep で確認する
- **既存パターン尊重** — 実装アプローチを問う際は、既存コードベースの規約に従う選択肢を必ず含める
- **完遂するまで続ける** — 実装者が追加質問なしに作れるレベルになるまで質問を続ける。早期終了しない。終わったと思ったら spec をもう一度通読し、まだ曖昧・不十分な箇所を探す。インタビューは長く徹底的になる前提で臨む

## Current Status セクション

spec の末尾には常に「Current Status」セクションを置き、インタビューの進捗を記録する。

**インタビュー中**（簡潔に保つ）:
- カバー済みの領域と、そこで下した決定の要約
- 残りの領域と、概算の残質問数

例:
```
## Current Status

Interview in progress.

Covered:
- Data Model: schema fields finalized, index fixed to non-unique
- Architecture: service object pattern, module structure decided

Remaining (~12 questions):
- Behavior & Logic: flow details, error handling
- Security: rate limiting defaults, replay prevention
- Operations: setup, purge job
- Scope & Deliverables: docs file scope
```

**インタビュー完了後** は、実装者向けの Checklist + Updates 形式に書き換える:
- 各成果物のチェックボックス付き実装チェックリスト
- 日付付き進捗エントリを記録する Updates サブセクション
- 冒頭に指示文を置く: "Implementors MUST keep this section updated as they work."

## 関連コマンドとの使い分け

- `/spec-interview` — 短時間で設計判断のトレードオフだけを深掘りする軽量版。spec 本体の網羅性ではなく「設計判断の確定」にフォーカス
- `/spec-review` （このコマンド） — 参照ファイル・ロードマップ・外部ライブラリ仕様まで踏み込んで網羅的に確認し、実装者が spec 単独で動けるレベルまで磨き上げる exhaustive 版
- `/spec` — Kiro 式 spec 駆動開発（要件・設計・タスクを 3 ファイル構成で管理）
- `/plan` — spec を書かず直接実装計画を立てる
