## Spec New

既存ファイル不要。**何を作るかの一言から始め**、AskUserQuestionTool を使った深掘りインタビューで仕様を完全に構築する。自明な質問は排除し、実装上のトレードオフ・エッジケース・非機能要件の具体的閾値など、設計判断に直結する質問のみを行う。インタビュー完了後、新しい spec ファイルを生成する。

### 使い方

```bash
/spec-new [出力ファイルパス]
```

- 引数なし: カレントディレクトリに `SPEC.md` を生成
- 引数あり: 指定パスにファイルを生成（例: `/spec-new docs/SPEC.md`）

### Claude が守るべき行動指針

#### 必須ルール

1. **必ず AskUserQuestionTool を使う** — テキストで質問を羅列するのではなく、毎回 `AskUserQuestionTool` を呼び出して選択肢付きの質問を提示する
2. **1 回に 2〜4 問** — 一度に大量の質問をせず、回答を受けてから次の深掘りをする
3. **Phase 0 以降は自明な質問を禁止** — 対象・目的が判明した後は「何を作りますか？」「誰が使いますか？」といった質問は絶対にしない
4. **回答を踏まえた follow-up** — 各回答に対して「その選択をした場合の具体的な問題」を掘り下げる
5. **完全になるまで継続** — インタビューを途中で切り上げず、全カテゴリが網羅されるまで続ける
6. **最後に必ず書き込む** — インタビューが完了したら、Write ツールで spec ファイルを新規作成する

#### 質問の質の基準

**✅ 良い質問の例**
- 「A と B のアプローチを比較したとき、あなたが優先したいのはどちら？その理由は？」
- 「〇〇 が失敗したとき、ユーザーはどのような状態になることを許容できる？」
- 「レスポンスタイムの上限は具体的に何 ms？99 パーセンタイルで？」
- 「1 年後にこの機能をどう変える想定？その拡張を今から考慮すべき？」
- 「A を優先すると B のパフォーマンスが 30% 落ちる可能性がある。それでも OK？」
- 「実装で一番難しいと予想している部分はどこ？」
- 「MVP として最初に外すべき機能はどれ？」

**❌ 悪い質問の例**（Phase 0 以降は禁止）
- 「この機能の目的は何ですか？」（Phase 0 で確認済みのはず）
- 「認証は必要ですか？」（対象が判明してから深掘りすべき）
- 「ユーザーは誰を対象にしますか？」（Phase 0 で確認済みのはず）

### インタビューフェーズ

#### Phase 0: 対象把握（最初の 1 ラウンドのみ）

まだ何も分かっていないため、ここだけは「何を作るか」を確認する質問を許可する。**ただし 1 ラウンドで終わらせ、得た情報を元にすぐ深掘りに移行する。**

AskUserQuestionTool で以下を聞く:
- 「何を作りたいか」（自由記述 or 選択肢があれば提示）
- 「主要ユーザーは誰か・どんな場面で使うか」
- 「現時点で決まっている技術スタックはあるか」

#### Phase 1: 深掘りインタビュー（ループ）

Phase 0 の回答を踏まえ、以下のカテゴリを網羅するまでインタビューを繰り返す。**各ラウンドで 2〜4 問**を AskUserQuestionTool で提示し、回答を受けて次のラウンドに進む。

**必須カテゴリ（全て確認すること）**

- **コア機能の優先順位**: MVP に含める機能と後回しにする機能、最初に動かすべき最小単位
- **技術アーキテクチャ**: ライブラリ・フレームワーク選択のトレードオフ、データモデルの具体的構造、API 設計の詳細
- **エラー・失敗シナリオ**: 各操作の失敗時の挙動、部分的失敗のハンドリング、リトライ戦略
- **非機能要件の数値**: レイテンシ閾値、スループット要件、可用性 SLA、データ保持期間
- **セキュリティ**: 認証・認可の詳細、機密データの扱い、脅威モデル
- **スケーラビリティ**: 現在の想定規模、将来の成長率、ボトルネック予測
- **UI/UX の具体的挙動**: 非同期処理中の UI 状態、エラー表示の詳細、モバイル・アクセシビリティ対応
- **データの整合性**: 競合状態のハンドリング、トランザクション境界、キャッシュ戦略
- **運用・監視**: ログに含める情報、アラート条件、デプロイ戦略
- **依存関係とリスク**: 外部サービス依存のリスク、最も不確実な部分、想定外のシナリオ

**AskUserQuestionTool の使い方**

各質問では必ず以下の形式を心がける:
- 選択肢は「具体的な技術名・数値・設計パターン」を提示する（「その他」は必要に応じて）
- 選択肢の説明でトレードオフを簡潔に示す
- 「Other」で自由記述を許容する

#### Phase 2: 完了判定

以下の全条件を満たしたらインタビュー完了と判断する:

- [ ] 全必須カテゴリをカバーした
- [ ] 主要なエラーシナリオが定義された
- [ ] 非機能要件に具体的な数値がついた
- [ ] 技術的な重要決定が確定した
- [ ] 未解決の曖昧さが spec に影響しない軽微なものだけになった

#### Phase 3: Spec 新規作成

収集した情報をもとに spec ファイルを新規作成する:

1. 以下のセクション構成で書く:
   - **概要**: 何を作るか・対象ユーザー・解決する問題
   - **コア機能**: MVP スコープと優先順位付き機能一覧
   - **技術スタック・アーキテクチャ**: 選択した技術とその根拠、アーキテクチャ構成
   - **データモデル**: 主要エンティティと関係
   - **API 仕様**: 主要エンドポイントとリクエスト/レスポンス形式
   - **エラーハンドリング**: 失敗ケース一覧と対応方針
   - **非機能要件**: レイテンシ・可用性・スループット等の数値付き要件
   - **セキュリティ要件**: 認証・認可・データ保護の方針
   - **UI/UX 仕様**: 主要フロー・状態遷移・エラー表示
   - **運用・監視**: ログ戦略・アラート・デプロイ方針
   - **未解決の課題・将来の拡張候補**: 今回スコープ外にした判断とその理由
2. Write ツールでファイルに書き込む

### 実行フロー例

```
/spec-new

[Phase 0 - Round 1] AskUserQuestionTool:
  - Q1: 何を作りたいですか？（自由記述）
  - Q2: 主なユーザーはどんな人で、どんな場面で使う？
  - Q3: 決まっている技術スタックはある？

[ユーザー回答: 「チームの作業ログを共有する Web アプリ。エンジニア 10 人、毎日使う。React + Node.js 予定」]

[Phase 1 - Round 1] AskUserQuestionTool（深掘り開始）:
  - Q1: 作業ログの「共有」はリアルタイム同期 vs 手動更新、どちらが重要？
  - Q2: ログのフォーマットは自由記述 vs 構造化フォーム（カテゴリ・所要時間等）、どちら？

[ユーザー回答]

[Phase 1 - Round 2] AskUserQuestionTool:
  - Q1: リアルタイム同期を選んだ場合、10 人が同時編集した際の競合はどう解決する？
  - Q2: ログの検索・フィルター機能は MVP に含める？含める場合、最低限必要な絞り込み条件は？
  - Q3: ログデータの保持期間はどれくらいを想定する？

... （完了まで継続）

→ 全カテゴリ完了
→ Write ツールで SPEC.md を新規作成
→ 「インタビュー完了。SPEC.md を作成しました。」
```

### 注意事項

- **前提条件**: なし（ファイル不要でゼロから始められる）
- **制限事項**: AskUserQuestionTool の選択肢は最大 4 つ。複雑な選択は複数ラウンドに分割する
- **推奨事項**: インタビューの途中でユーザーが「終わりにして」と言ったら、それまでの情報で spec を書き込む
- **上書き確認**: 出力先にすでにファイルが存在する場合は上書きしてよいか確認する

### 関連コマンド

- `/spec-interview` : 既存の spec ファイルを読み込んで深掘りする版
- `/spec` : Kiro 式 spec 駆動開発（要件・設計・タスクを 3 ファイルで管理）
- `/plan` : 実装計画の作成（spec なしで直接計画を立てる場合）
